# 团体赛报名与导出设计文档

## 1. 背景

当前团体赛流程已具备领队申请、邀请队友、指定单打、队伍报名等基础能力，但在以下方面存在明显问题：

- 队伍草稿与正式报名状态混在一起，导致前台赛事详情、后台报名管理、导出报名表的数据不清晰。
- 团体赛人数、性别、单打人数缺少统一的赛事级配置，难以支撑男子团体、女子团体、混合团体等不同赛事。
- 领队在组队阶段缺少删除、替换、取消邀请等必要操作，一旦邀请错误或性别不匹配，流程容易卡死。
- 导出报名表缺少稳定排版，无法同时清晰表达队伍信息和成员明细。

本设计文档用于固化团体赛报名规则，作为后续开发与验收的统一依据。

## 2. 目标

- 明确团体赛从组队到正式报名的状态边界。
- 支持按赛事配置每队人数上限、人数下限、性别规则、单打人数。
- 保证未正式提交的队伍不进入前台赛事详情、后台报名列表和导出数据。
- 支持领队在正式提交前删除、替换队员并处理待响应邀请。
- 统一团体赛单打名单的锁定规则与积分规则。
- 设计一张就足够清晰的报名导出表。

## 3. 范围

本次设计覆盖：

- 后台团体赛配置项
- 小程序团体赛报名流程
- 邀请状态与名额占用规则
- 人数/性别/单打校验规则
- 前台赛事详情展示边界
- 后台报名管理展示边界
- 导出报名表结构
- 订阅消息通知规则

本次设计不覆盖：

- 正式提交后的修改能力
- 用户自行修改注册性别
- 团体赛对阵编排与出场顺序编排

## 4. 关键定义

### 4.1 队伍状态

- `草稿`：领队已开始组队，但尚未正式提交报名。
- `已提交`：领队已完成最终提交，队伍正式进入报名名单。

### 4.2 邀请状态

- `待处理`：队员已收到邀请，但未作出选择。
- `已加入`：队员接受邀请并加入队伍。
- `已拒绝`：队员明确拒绝邀请。
- `已取消`：领队在对方处理前取消邀请。

### 4.3 名额与人数口径

- `占用名额` = 参赛领队 + 已加入队员 + 待处理邀请
- `实际参赛人数` = 参赛领队 + 已加入队员
- `单打名单` = 从实际参赛人数中指定的固定人数名单

说明：

- 人数规则、性别规则、单打规则，统一按`实际参赛人数`校验。
- 邀请上限控制使用`占用名额`。

## 5. 方案取舍

### 5.1 队伍草稿展示方案

方案 A：草稿队伍在前台和后台都展示，并标记未提交。

- 优点：可观察组队进度。
- 缺点：前后台数据噪音大，容易被误认为已报名。

方案 B：草稿队伍只在领队自己的组队页面显示。

- 优点：边界最清晰，正式报名名单不会混入草稿。
- 缺点：后台无法直接查看未提交草稿。

结论：采用方案 B。

### 5.2 提交后可编辑方案

方案 A：正式提交后直到截止时间仍允许修改。

- 优点：更灵活。
- 缺点：状态和校验复杂，后台和导出会反复变化。

方案 B：正式提交后立刻锁死。

- 优点：规则简单，导出和后台口径稳定。
- 缺点：领队提交前必须确认清楚。

结论：采用方案 B。

### 5.3 导出表方案

方案 A：一队一行，成员名单和单打名单写入单元格文本。

- 优点：表很短。
- 缺点：成员明细难筛选，后续统计不方便。

方案 B：一人一行，队伍信息每行重复。

- 优点：最利于筛选。
- 缺点：同一队信息重复较多，阅读感受一般。

方案 C：单表导出，按成员逐行展开，队伍级字段按队伍分组合并显示。

- 优点：既能看队伍整体，也能看成员明细；一张表即可完成查看和筛选。
- 缺点：导出实现需要处理分组或合并单元格。

结论：采用方案 C。

## 6. 后台赛事配置

团体赛配置项按赛事独立设置，不做全局固定规则。

### 6.1 必填配置项

- 参赛队伍上限
- 每队最少人数
- 每队最多人数
- 性别规则
- 每队单打人数

### 6.2 性别规则

支持以下模式：

- `不限`
- `仅男子`
- `仅女子`
- `固定男 X / 女 Y`
- `至少男 X / 女 Y`

其中：

- `固定男 X / 女 Y`：实际参赛男生人数必须等于 X，女生人数必须等于 Y。
- `至少男 X / 女 Y`：实际参赛男生人数必须大于等于 X，女生人数必须大于等于 Y。

### 6.3 后台配置校验

- 最少人数必须大于等于 2。
- 最多人数必须小于等于 16。
- 最少人数不能大于最多人数。
- 每队单打人数必须大于等于 1，且不能大于每队最多人数。
- `固定男 X / 女 Y` 时，`X + Y` 必须位于最少/最多人数范围内。
- `至少男 X / 女 Y` 时，`X + Y` 不能大于每队最多人数。

## 7. 前端报名流程

### 7.1 基本流程

1. 用户申请成为某赛事领队。
2. 审核通过后进入组队页。
3. 领队必须先填写`队伍名称`，才能开始邀请。
4. 领队选择自己是否参赛。
5. 领队邀请队员加入队伍。
6. 队员接受或拒绝邀请。
7. 领队在正式提交前可调整队名、成员、单打名单。
8. 校验通过后，领队执行最终提交。
9. 提交成功后队伍状态变为`已提交`，并立即锁死。

### 7.2 领队在正式提交前可执行的操作

- 修改队伍名称
- 选择自己是否参赛
- 邀请队员
- 取消待处理邀请
- 删除已加入队员
- 替换队员
- 指定单打名单

### 7.3 正式提交后锁定内容

- 队伍名称
- 领队参赛状态
- 队员名单
- 单打名单

正式提交后不允许：

- 删除成员
- 替换成员
- 取消邀请
- 修改队名
- 修改单打名单

## 8. 邀请与名额规则

### 8.1 邀请入口

- 领队未填写队名时，不允许发起邀请。
- 达到队伍人数上限时，不隐藏邀请入口。
- 邀请入口改为禁用状态，并提示：

`已达队伍人数上限，请先删除队员或取消邀请`

### 8.2 队员响应

邀请页面至少提供两个按钮：

- `加入队伍`
- `暂不加入 / 拒绝`

规则如下：

- 队员不操作，状态保持`待处理`，不算拒绝。
- 队员明确点击拒绝，状态变为`已拒绝`。
- 队员接受后，状态变为`已加入`。

### 8.3 名额占用与释放

- `待处理邀请`占用名额。
- `已加入队员`占用名额。
- 只有以下情况释放名额：
  - 队员明确拒绝邀请
  - 领队取消待处理邀请
  - 领队删除已加入队员

### 8.4 待处理邀请与提交关系

- 存在`待处理邀请`时，不允许正式提交报名。
- 领队必须先将待处理邀请处理为：
  - 队员已拒绝
  - 领队已取消
  - 队员已加入

## 9. 领队参赛规则

- 领队是否参赛由领队本人在前端决定，不作为后台赛事配置项。
- 领队选择`参赛`时：
  - 计入实际参赛人数
  - 计入性别统计
  - 可被选择进入单打名单
- 领队选择`不参赛`时：
  - 不计入实际参赛人数
  - 不计入性别统计
  - 不能进入单打名单

当领队切换参赛状态时，系统应立即重算：

- 实际参赛人数
- 男女人数
- 单打名单是否仍然有效

若领队原本已被选入单打名单，随后改为不参赛，则需自动移除其单打资格并提示领队重新确认名单。

## 10. 人数与性别校验

### 10.1 数据来源

- 注册时必须填写性别。
- 团体赛性别校验统一使用注册用户性别字段。
- 用户前端不可自行修改性别。

### 10.2 邀请对象

已注册用户：

- 接受邀请时，后台根据其注册性别进行校验。

未注册用户：

- 先完成注册并填写性别。
- 再进入邀请处理流程。
- 接受邀请时由后台执行校验。

### 10.3 校验时机

前端可做提示性校验：

- 当前实际参赛人数
- 当前男女人数
- 距离最低人数还差多少
- 性别规则是否满足
- 单打人数是否已选满

后台必须做硬校验，校验发生在：

- 接受邀请时
- 删除/替换成员后
- 领队切换是否参赛后
- 最终提交时

### 10.4 最终提交校验

正式提交时，后台必须校验：

- 实际参赛人数不小于最少人数
- 实际参赛人数不大于最多人数
- 性别规则满足赛事配置
- 单打名单人数等于赛事配置值
- 单打名单成员全部来自实际参赛人员
- 同一用户不能在同一赛事重复参赛

说明：

- 即使队伍名额已满，仍然可能因性别不达标而无法提交。
- 这种情况下，领队只能在提交前通过删除、替换队员来调整。

## 11. 单打名单与积分规则

### 11.1 单打名单

- 单打名单由领队在报名时一次性指定。
- 每支队伍的单打人数由赛事配置统一确定。
- 同一赛事中，所有队伍使用相同的单打人数规则。
- 单打名单只能从实际参赛人员中选择。

### 11.2 锁定规则

- 队伍正式提交后，单打名单立刻锁死。
- 锁死后不允许修改单打名单。

### 11.3 积分规则

- 团体赛中的单打场次计入个人积分。
- 积分胜负变化与普通单打完全一致。
- 同时计入：
  - 个人胜负记录
  - 历史战绩
  - 积分变化记录

## 12. 前台展示边界

### 12.1 领队自己的组队页

可见：

- 草稿队伍
- 队名
- 领队参赛状态
- 当前成员
- 待处理邀请
- 单打名单选择
- 提交按钮

### 12.2 赛事详情页

仅展示`已提交`队伍。

不展示：

- 草稿队伍
- 未正式提交的成员信息
- 未处理邀请

赛事详情中的参赛队伍建议展示：

- 队伍名称
- 领队姓名
- 领队是否参赛
- 实际参赛人数
- 队员名单
- 单打名单

## 13. 后台报名管理边界

后台报名管理仅展示`已提交`队伍。

不展示：

- 草稿队伍
- 未处理邀请
- 未正式提交成员

后台列表建议字段：

- 队伍名称
- 领队姓名
- 领队是否参赛
- 实际参赛人数
- 男女人数
- 单打名单
- 提交时间

后台详情建议字段：

- 队伍名称
- 领队信息
- 队员明细
- 性别构成
- 单打名单
- 报名校验结果快照

## 14. 通知规则

已具备订阅消息能力时，建议支持以下通知：

- 领队发出邀请后，通知被邀请人
- 队员接受邀请后，通知领队
- 队员拒绝邀请后，通知领队
- 领队取消待处理邀请后，通知被邀请人
- 领队删除已加入队员后，通知被移出成员
- 队伍正式报名成功后，通知全体已加入成员

建议消息文案：

- `你收到某某队的组队邀请`
- `某某已接受你的组队邀请`
- `某某已拒绝你的组队邀请`
- `领队已取消对你的组队邀请`
- `领队已将你移出队伍`
- `你的队伍已完成赛事报名`

## 15. 导出报名表设计

### 15.1 导出目标

在一个工作表内同时满足：

- 能清楚看出每支队伍的整体信息
- 能清楚看出每个队员的明细信息
- 能快速筛选单打队员
- 能快速核对男女构成和参赛人数

### 15.2 推荐结构

采用`单表导出 + 按成员逐行展开 + 队伍字段按队伍分组显示`。

含义如下：

- 一支队伍占用多行，每个实际参赛成员一行。
- 队伍级字段可在 Excel 中合并单元格显示，也可在导出时重复写值。
- 如后续考虑筛选兼容性，优先选择“不合并、逐行重复队伍字段”；如考虑阅读美观，可在下载后的模板里做分组合并。

### 15.3 表头设计

建议列顺序如下：

| 列名 | 说明 |
|------|------|
| 序号 | 报名队伍顺序号 |
| 队伍名称 | 队伍名称 |
| 领队姓名 | 队伍领队 |
| 领队是否参赛 | 是 / 否 |
| 实际参赛人数 | 最终报名人数 |
| 性别构成 | 例如 `男2 / 女2` |
| 单打人数 | 赛事要求的单打人数 |
| 单打名单 | 该队单打人员汇总 |
| 成员序号 | 队内第几位成员 |
| 成员姓名 | 实际参赛成员姓名 |
| 成员性别 | 男 / 女 |
| 成员身份 | 领队 / 队员 |
| 是否单打 | 是 / 否 |
| 手机号 | 便于线下联系 |
| 提交时间 | 最终报名提交时间 |

### 15.4 排版规则

- 同一队伍的成员行连续排列，不允许被其他队伍打断。
- 每支队伍的第一行使用浅底色或加粗，便于视觉分组。
- 队伍切换处加细边框或空白分隔行，提升阅读性。
- `单打名单`列展示为队内名单汇总，例如：`张三、李四、王五`
- `是否单打`列用于逐成员核对。
- `领队是否参赛`和`成员身份`一起使用，可明确看出领队是否在实际报名名单中。

- 已取消邀请

## 16. 异常与提示文案

建议在前端和后台使用统一文案：

- 未填写队伍名称：`请先填写队伍名称，再邀请队员`
- 达到人数上限：`已达队伍人数上限，请先删除队员或取消邀请`
- 存在待处理邀请：`还有待处理邀请，请先确认或取消后再提交`
- 性别不符合：`当前队伍性别人数不符合赛事要求`
- 单打人数不足：`请按赛事要求选择单打队员`
- 单打人选非法：`单打队员必须来自实际参赛名单`
- 提交成功：`报名成功，队伍信息已锁定`

## 17. 测试与验收重点

### 17.1 组队流程

- 未填写队名时无法邀请
- 填写队名后可邀请
- 达到上限后邀请入口禁用而不是消失
- 取消待处理邀请后名额释放
- 删除已加入队员后名额释放
- 可成功替换队员

### 17.2 校验规则

- 领队参赛时计入人数与性别
- 领队不参赛时不计入人数与性别
- 性别规则按实际参赛人数校验
- 存在待处理邀请时不可提交
- 单打名单人数不符时不可提交

### 17.3 展示边界

- 草稿队伍仅在领队组队页可见
- 前台赛事详情仅显示已提交队伍
- 后台报名管理仅显示已提交队伍
- 导出仅包含已提交队伍

### 17.4 通知

- 发邀请通知正常
- 接受/拒绝后通知领队正常
- 取消邀请通知队员正常
- 删除已加入队员通知队员正常
- 报名成功通知全员正常

## 18. 最终结论

本设计确定以下核心规则：

- 领队必须先填写队名，才能邀请队员。
- 团体赛按赛事配置每队人数范围、性别规则、单打人数。
- 领队是否参赛由前端选择，参赛时需计入人数与性别。
- 待处理邀请占用名额，不操作不算拒绝。
- 只有拒绝、取消邀请、删除成员才释放名额。
- 草稿队伍不进入前台赛事详情、后台报名和导出。
- 正式提交后队名、成员、单打名单立刻锁死。
- 团体赛单打计入个人积分，规则与普通单打一致。
- 报名导出采用单表模式，按成员逐行展开，兼顾队伍信息与成员明细。
